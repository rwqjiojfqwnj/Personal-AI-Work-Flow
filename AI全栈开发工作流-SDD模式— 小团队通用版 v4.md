# AI 全栈开发工作流（SDD 模式）— 小团队通用版 v4

> **适用场景**：Vue 3 + TypeScript + Java / Spring Boot 前后端分离，小团队协作（2-8 人）或一人全栈  
> **核心目标**：兼顾企业开发中的通用性、实用性、可恢复性、可验收性  
> **核心理念**：OpenSpec 定义 WHAT → Superpowers 设计 HOW → Comet 编排阶段 → codegraph 理解现状与影响 → Impeccable 提升前端视觉质量 → 你负责最终判断与验收

---

## 1. 先说结论

这套工作流不是让 AI 代替研发流程，而是让 AI 在企业开发的关键节点稳定发挥：

1. **需求先澄清，再设计，再并行开发**
2. **Spec、契约、验收口径必须先对齐，不能边做边猜**
3. **AI 负责提速，事实源和最终验收仍由团队掌控**
4. **工具分工明确，不让一个工具承担全部职责**

---

## 2. 工具定位

### 2.1 工具职责分工

| 工具           | 定位      | 负责什么                                              | 不负责什么       |
| ------------ | ------- | ------------------------------------------------- | ----------- |
| OpenSpec     | WHAT    | 需求变更、proposal、spec、tasks、归档                       | 技术实现细节      |
| Superpowers  | HOW     | brainstorming、design、plan、TDD、review、subagent     | 需求事实源管理     |
| Comet        | 阶段编排    | open → design → build → verify → archive 的状态流转与守护 | 业务规则本身      |
| codegraph    | 分析层     | 理解代码结构、依赖关系、影响范围、diff 分析                          | 需求事实源、契约事实源 |
| Impeccable   | 前端设计质量层 | UI/UX 设计、样式审查、视觉迭代、前端反模式检测                        | 全栈研发主流程     |
| Element Plus | 前端基础设施  | 稳定的 Vue 组件能力、表单表格弹窗等基础控件                          | 项目流程治理      |

### 2.2 企业项目中的推荐组合

- **主流程**：`Comet + OpenSpec + Superpowers + codegraph`
- **前端设计增强**：`Impeccable`
- **组件库默认基线**：`Element Plus`

| 阶段 | 名称 | 核心目标 | 主要工具 |
|------|------|----------|----------|

## 5. 阶段 0：项目基线初始化

### 5.1 目标

让项目具备“可被 AI 稳定理解和执行”的最小基础环境。

### 5.2 必做事项

1. 初始化 `Comet + OpenSpec + Superpowers + codegraph`
2. 确定前端组件库：默认固定为 `Element Plus`
3. 决定是否引入 `Impeccable` 作为前端设计质量增强
4. 建立 `docs/workflow-templates/` 基线模板目录
5. 建立项目知识图谱

### 5.3 推荐动作

```text
1. 安装并初始化 comet
2. 准备项目目录结构
3. 建立 workflow templates
4. 执行 /understand --language zh --full
5. 确认前后端基础项目可启动
```

### 5.4 codegraph 的定位

`codegraph` 在企业项目里应被视为：

- **现状理解工具**
- **影响分析工具**
- **辅助评审工具**

不应被视为：

- 需求事实源
- 契约事实源
- 是否上线的唯一判断依据

### 5.5 阶段退出条件

- 工具初始化完成
- 项目基线模板已建立
- 组件库已选定
- 图谱已建立
- 前后端项目可运行

## 6. 阶段 1：需求澄清与变更立项

**建议命令入口**：`/comet-open`

### 6.1 目标

在编码前，先把以下问题澄清：

- 这次到底做什么
- 不做什么
- 依赖什么外部接口或外部系统
- 风险在哪里
- 验收时看什么

### 6.2 主要输入

- PRD / 业务描述
- 外部接口文档
- 现有系统约束
- codegraph 图谱

### 6.3 主要输出

- `proposal.md`
- `spec.md`
- `tasks.md`
- 需求澄清清单
- 风险清单
- 初版验收范围

### 6.4 企业场景下必须明确的问题

1. 本期最小闭环是什么
2. 哪些属于后续迭代
3. 哪些功能依赖三方接口
4. 权限和角色边界是什么
5. 错误场景和兜底策略是什么
6. 对页面、接口、数据、日志、审计是否有额外要求

### 6.5 推荐输出结构

- **业务目标**
- **范围边界**
- **外部依赖**
- **风险项**
- **待确认问题**
- **验收关注点**

### 6.6 阶段退出条件

- 变更已立项
- 范围清晰
- 风险已识别
- 未决问题已分类
- OpenSpec change 已可供后续设计使用

---

## 7. 阶段 2：深度设计与契约冻结

**建议命令入口**：`/comet-design`

这是最关键的阶段。  
企业里返工大多不是因为“不会写代码”，而是因为“设计没有冻结就开始并行”。

### 7.1 统一设计必须覆盖的内容

1. 业务流程
2. 数据结构
3. API 设计
4. 权限模型
5. 异常与错误码
6. 异步任务与状态机
7. 前后端契约
8. 验收标准

### 7.2 单一契约源原则

必须明确一条规则：

> **接口契约必须以 OpenAPI 作为单一事实源。**

同步顺序：

1. `OpenAPI` 作为单一契约源
2. 前端 `types`
3. 后端 `DTO / VO`
4. Mock 数据
5. Swagger 展示

前端类型、后端 DTO、Mock、接口文档不能长期各写各的。

### 7.3 角色化设计

在统一设计完成后，再进入角色化设计拆分。建议最少按以下 4 个视角拆：

#### 1. 产品/业务视角

- 业务闭环是否完整
- 是否覆盖关键边界条件
- 是否有流程中断点
- 验收口径是否可执行

#### 2. 后端设计视角

- 领域模型
- 接口契约
- 权限、事务、幂等、缓存
- 三方接口适配
- 异步任务与日志审计

#### 3. 前端设计视角

- 页面流转
- 状态管理
- 组件拆分
- 交互反馈
- 空状态、错误状态、加载状态

#### 4. 测试/验收视角

- 验收矩阵
- 接口测试策略
- 页面回归路径
- 高风险边界场景

### 7.4 前端 UI 与样式策略

#### 7.4.1 AI 能不能直接写样式

可以，但只建议在以下边界内直接写：

- 基于成熟组件库的业务页
- 列表页、详情页、搜索页、表单页、弹窗页
- 已有品牌和主题规范下的样式补充

不建议完全裸写：

- 整套设计系统
- 营销型首页视觉探索
- 高品牌辨识度视觉语言

#### 7.4.2 组件库固定策略

本工作流默认将组件库固定为 `Element Plus`。

固定原因：

- 更适合大多数企业中后台场景
- 表单、表格、树、弹窗等基础能力成熟
- 团队协作时语义和组件共识更稳定
- 能降低 AI 在组件选型上的随机性

如项目已有成熟设计系统或历史包袱，不强制迁移，但新项目默认按 `Element Plus` 组织。

#### 7.4.3 Impeccable 适合什么时候接入

建议在以下场景使用：

1. 新建或重做重要页面时，先用 `/impeccable shape`
2. 页面完成后，用 `/impeccable critique`
3. 准备交付前，用 `/impeccable polish`
4. 质量回归时，用 `/impeccable audit`
5. UI 文件持续改动较多时，启用 detector / hook

### 7.5 多 Agent 何时允许启动

只有当以下条件都满足，才建议开多个 Agent 并行：

1. 统一设计已确认
2. 单一契约源已明确
3. 角色化设计已完成
4. 验收标准已有初版
5. 风险和边界已记录

### 7.6 阶段退出条件

- 统一设计完成
- 契约冻结
- 角色化设计完成
- 验收矩阵初版完成
- 可以进入计划拆分

## 8. 阶段 3：计划、构建与联调

**建议命令入口**：`/comet-build`

### 8.1 目标

把设计转为可执行任务，并在不破坏一致性的前提下尽量并行。

### 8.2 多 Agent 协作规则

#### 单 Agent

适合：

- 小需求
- 单模块修改
- 高耦合逻辑

#### 双 Agent

适合：

- 标准前后端分离功能
- 契约稳定

建议分工：

- `frontend-agent`
- `backend-agent`

#### 三 Agent

适合：

- 中等复杂功能
- 前后端之外还需要明确验收/测试收口

建议分工：

- `frontend-agent`
- `backend-agent`
- `qa-agent`

#### 禁止并行的情况

- 需求仍在频繁变动
- 契约未冻结
- 三方接口能力不确定
- 核心领域模型还在变化

### 8.3 实现顺序建议

#### 模式 A：后端骨架优先，前端并行 Mock

推荐默认模式。

1. 后端先给出 API 骨架和统一返回结构
2. 单一契约源同步到前端类型
3. 前端使用 Mock 开发交互
4. 后端逐步补齐真实实现
5. 联调时替换 Mock

#### 模式 B：设计冻结后的真正并行

适用于：

- 契约稳定
- 团队配合成熟
- 需求边界清晰

### 8.4 前端样式开发建议

企业项目里，建议按以下顺序：

1. 先选组件库
2. 再定主题 token
3. AI 写页面级布局和样式
4. Impeccable 做 critique / polish / audit
5. 浏览器实测

### 8.5 联调要求

- 不只看“页面能打开”
- 要对照验收矩阵逐项联调
- 记录接口偏差、样式偏差、交互偏差

### 8.6 阶段退出条件

- 开发任务完成
- 联调通过
- tasks 状态已更新
- 关键文档同步完成

## 9. 阶段 4：验证、验收与发布准备

**建议命令入口**：`/comet-verify`

### 9.1 验证分 4 层

#### 1. 静态与单元验证

- 后端：`mvn test`
- 前端：`lint`、`build`
- 必要时补关键单测 / 集成测试

#### 2. 契约一致性验证

- 单一契约源与前端 types 是否一致
- 单一契约源与后端 DTO 是否一致
- Mock 是否过期

#### 3. 联调与回归验证

- 关键接口联调
- 页面主流程回归
- 空状态、错误状态、权限场景验证

#### 4. 设计与视觉质量验证

- 重要页面运行 `/impeccable audit`
- 必要时对重点页面运行 `/impeccable polish`
- 检查组件库使用是否一致

### 9.2 验收矩阵必须落地

不要只写“已完成”，而要能回答：

- 哪个需求点已经完成
- 对应哪个接口
- 对应哪个页面
- 被哪个测试或验收动作验证过
- 结果是什么

### 9.3 codegraph 在本阶段的作用

验证阶段的 `codegraph` 主要用于：

- 分析本次变更影响范围
- 对大改动做回归提示
- 协助评审遗漏点

### 9.4 阶段退出条件

- 验证报告存在
- 验收矩阵已更新
- 风险项已说明
- 可进入归档

---

## 10. 阶段 5：归档与复盘

**建议命令入口**：`/comet-archive`

### 10.1 必做事项

1. 将 delta spec 同步到主 spec
2. 更新 change 状态
3. 补充开发总结
4. 更新图谱
5. 记录遗留问题和技术债

### 10.2 复盘至少回答 4 个问题

1. 这次做了什么
2. 哪些设计决策是有效的
3. 哪些地方返工了，为什么
4. 哪些规则应该沉淀到项目基线模板

### 10.3 阶段退出条件

- 归档完成
- 图谱更新完成
- 复盘完成

## 11. 常用决策速查

| 问题 | 建议 |
|------|------|
| 新项目第一步做什么？ | 初始化 comet 和项目基线，建立 codegraph 图谱 |
| PRD 来了先干什么？ | `/comet-open` 做需求澄清与立项 |
| 什么时候开多个 Agent？ | 统一设计、契约、验收口径冻结后 |
| 前端样式让 AI 直接写吗？ | 可以，但建议基于 Element Plus 写业务样式 |
| 前端设计增强用什么？ | 重要页面接入 Impeccable |
| codegraph 是不是事实源？ | 不是，它是理解和影响分析工具 |
| 契约以什么为准？ | 固定以 OpenAPI 为单一契约源 |
| 验收怎么做？ | 必须落验收矩阵，不只跑命令 |

## 12. 快速使用建议

### 12.1 新项目初始化

```text
1. 初始化 comet / openspec / superpowers / codegraph
2. 组件库固定为 Element Plus
3. 重要前端项目可接入 Impeccable
4. 复制 workflow templates
5. 建立项目图谱
```

### 12.2 新需求进入

```text
1. /comet-open
2. 澄清需求和边界
3. 建立 change / spec / tasks
4. 输出风险和待确认问题
```

### 12.3 设计阶段

```text
1. /comet-design
2. 统一设计
3. 角色化设计
4. 契约冻结
5. 重要前端页面可先用 /impeccable shape
```

### 12.4 构建阶段

```text
1. /comet-build
2. 根据条件选择单 Agent / 多 Agent
3. 前后端并行
4. 重要页面做 Impeccable critique / polish
```

### 12.5 验证阶段

```text
1. /comet-verify
2. 测试、契约校验、联调、验收矩阵
3. UI 页面执行 /impeccable audit
4. 通过后归档
```

## 13. 与 v3 相比的核心变化

1. 从“个人提示词流程”升级为“小团队通用流程”
2. 把 `OpenSpec / Superpowers / Comet / codegraph / Impeccable` 的职责拆清楚
3. 增加“OpenAPI 作为单一契约源”的原则
4. 增加“角色化设计”与“多 Agent 并行规则”
5. 增加“前端 UI 与样式策略”
6. 增加“验收矩阵”而不是只做命令验证
7. 把项目约束拆到模板，避免主文档臃肿
