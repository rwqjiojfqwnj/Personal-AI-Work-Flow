设计时需要遵守以下约束：

# 【后端性能约束】
## 1.数据建模约束

- 状态字段优先用数字，不用英文字符串。
- 删除默认使用逻辑删除，不做物理删除，尤其是同步记录表。
- 表结构尽量收敛，不要为了状态机无节制加字段；能合并表达的就合并。
- 每条同步 / 异步任务数据都应至少具备：
    - 主业务 ID
    - 外部文档 ID（如 `ragFlowId`）
    - 当前任务类型
    - 当前状态
    - 重试次数
    - 下次重试时间
    - 错误码或错误信息
    - 创建时间、修改时间
    - 逻辑删除标记

## 2 查询与性能约束

- 能批量查询就不要循环查数据库。
- 能一次回表补齐数据就不要在组装 VO 时逐条补查。
- 对十几万级数据量默认按“批处理 + 分页/分批”设计，不能假设全量扫描压力很小。
- 列表查询必须分页，pageSize 上限 200
- 超过 3 张表的关联查询，拆分为多次查询在 Service 层组合
- TEXT/LONGTEXT 字段不参与列表查询，仅详情接口返回


## 3 缓存约束

- 数据库是真源，Redis 是缓存，不允许 Redis 成为唯一真源。
- 写操作以“数据库成功后删缓存”为主，不以“更新缓存”为主。
- 有缓存就必须考虑缓存失效、重建、并发击穿和旧版本结构兼容。
- 字典/配置/分类树等低频变更数据，用 Redis 缓存
- 缓存采用旁路缓存模式：先更新数据库，再删除缓存
- 禁止无过期时间的缓存
- 如果设计了缓存，必须同步说明：
    - key 规则
    - TTL
    - 写入时机
    - 删除时机
    - 重建来源
- 

## 4 其他约束
- 所有接口参数做 @Valid 校验
- 长时间任务必须异步，返回任务 ID 供前端轮询
- 调用三方接口记录请求和响应日志
- 异常日志包含完整堆栈
- 重要方法和关键步骤写注释
- 编码格式UTF-8

# 【前端性能约束】
- 列表必须分页加载，不做全量拉取
- 搜索输入做 300ms 防抖
- 相同接口并发请求去重
- 长列表用虚拟滚动
- 大表单 Tab 场景用 v-if 按需渲染
- 图片列表懒加载
- 所有异步操作有 loading 和 error 状态
- 表单提交按钮防重复点击
- 页面切换时未保存内容给离开确认

# 【前端 UI 约束】
- 组件库：[Element Plus / Ant Design Vue / Naive UI 三选一]
- 风格：简洁专业，间距 16px/24px，圆角 8px，合理留白
- 仅适配桌面端（≥1280px），不做移动端适配
